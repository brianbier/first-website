TDD is an approach is to first start thinking of the what we expect when the program works rather than the how we will get the problem to work. 
What all this means is that instead of trying to write the code that solves the problem we first define what the working code will do when it works and then we write the necessary code. 
 
The Best Approach is to start writing your test first and then stating your expectations of the code after since now we know our goal.
Example of a RSPEC Test
 
describe "fizzbuzz" do
  it 'returns "Fizz" when the number is divisible by 3' do
    fizz_3 = fizzbuzz(3)

    expect(fizz_3).to eq("Fizz")
  end
end
 
Use Describe.
The first thing RSpec allows you to do with its DSL is to define what it is you are describing. As in, are we writing a test that describes how our Billing System works or are we writing a test that describes how fizzbuzz works?
describe also accepts a block, which will be covered shortly, but for now, just know that a block in ruby starts with the dokeyword. Every block that is opened must be closed. We use the endkeyword to close the describe block.
Use IT
Every specification in RSpec begins with the itmethod
The it method, like the describe method, accepts an arbitrary string as an argument, namely, some text that says what we're currently interested in testing. Like all things in code, every it should be specifying one, only one, and exactly one, desired outcome or behavior. It's totally up to you, the programmer, to decide the phrasing for your itspecification, but a good starting point is a verb or action. it returns orit prints or it emails or it validates etc.
 
Expect, TO EQ
 
expect() is a method that accepts our unknown value or variable, the thing we're testing.  we chain a to()method to the expect(), Then finally we use what is known as a matcher, eq, to specify our expectation: that we expect the result of our expectation, passed to the expect method, to equal, to eq, 2. LIKE:
 
x = 1 + 1
expect(x).to eq(2)
